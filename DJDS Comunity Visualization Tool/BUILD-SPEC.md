## 0. How to use this document

This is the authoritative build spec. Build to it directly. Where it says **MUST**, treat it as a hard requirement; where it says **SHOULD**, use judgment but default to the stated behavior. Section 14 lists what is explicitly **out of scope** so you don't gold-plate.

Build order is in Section 13. Start there if you want to begin immediately.

Two notes before you start:
1. **This is a trauma-informed product for community members, many of whom are survivors of harm.** Every copy and design decision flows from that. Section 2 explains the framework — read it before writing UI. When in doubt, choose the calmer, gentler, lower-friction option.
2. **The visualization quality lives or dies by the Claude prompt** (Section 8). Budget real time there and expect to iterate against real output.

## 1. Product summary

A public-facing web app where community members describe their vision for a future space — at the scale of a **room, building, block, city, or planet** — by moving through a warm, sectioned, multi-step form. Their structured selections and open-text answers are assembled into a prompt and sent to the Anthropic Claude API, which returns a unique **SVG visualization** rendered in the browser.

Contributions can stand alone (shareable link, PNG export) or be collected into a **session** created by a DJDS facilitator. Facilitators get a dashboard with aggregate insights and a visibility toggle that controls whether contributors see a **collective mosaic** of everyone's visions.

**Primary users:** community members (non-technical, public-facing, mobile-heavy, multilingual).
**Secondary users:** DJDS facilitators running sessions.

## 2. Design principles (read first)

The tool is grounded in DJDS's trauma-informed design framework. These are not decoration — they shaped the question set and must shape the build.

**8 Trauma Transformation Needs:** Trust & Transparency · Empowerment · Cultural Relevance · Acknowledgement · Collaboration · Relationship · Settlement of the Body & Brain · Safety

**8 Sensory Systems:** Auditory (Hear) · Gustatory (Taste) · Olfactory (Smell) · Tactile (Touch) · Visual (See) · Vestibular (Balance) · Proprioception (Body Awareness) · Interoception (Internal Regulation)

What this means in practice:

- **Low friction, high agency.** No account creation. Almost nothing is required. The only hard gate in the whole flow is selecting a scale. Users can skip any question.
- **Calm by default.** No autoplay motion, no flashing, no urgency, no dark patterns. Transitions are short and quiet (≤150ms). Respect `prefers-reduced-motion`.
- **Embodied progression.** The flow deliberately moves: long-term vision → body/senses → belonging → sensory imagination → emotion → concrete features → narrative. Preserve this order.
- **Culturally generous.** The belonging question carries 46 identity/community options so people see themselves. Don't trim it for tidiness.
- **Transparent AI.** Tell users plainly that the image is generated by AI from their own words. Never imply a human drew it.
- **Mobile-first.** Must work well on a phone at a community event. Touch targets ≥ 44×44px.
- **Accessible.** Target WCAG 2.1 AA: semantic HTML, labeled inputs, visible focus rings, selection state conveyed by more than color (border + fill, not color alone).

Don't hardcode hex values throughout components — go through the tokens.

## 5. Full user flow

### 5.1 Community member flow
1. Land on the tool — either a standalone entry or a session link (`/s/[slug]`).
2. **Welcome screen** — 150-year vision check-in (Section 7.0).
3. **Step 1 — Scale** — pick one of five (Section 7.1). *This is the only required selection.*
4. **Step 2a — Sensory & Embodied Experience** — multi-select, no limit (7.2).
5. **Step 2b — Community & Belonging** — multi-select, no limit (7.3).
6. **Step 2c — Sensory Imagination** — open text, 500 chars (7.4).
7. **Step 3 — Feelings** — multi-select, **max 5** (7.5).
8. **Step 4 — Features** — scale-specific multi-select, **max 8** (7.6).
9. **Step 5 — Final Vision** — open text 800 chars + optional name (7.7).
10. **Generate** — submit to `/api/generate`, show loading state, render SVG.
11. **Vision page** (`/vision/[id]`) — view full-size, share link, export PNG, and (if in a session) it's been saved to that session.

### 5.2 Facilitator flow
1. Create a session at `/facilitator/new` (title, optional description, scale: fixed or "let participants choose", visibility default).
2. On creation, a **facilitator PIN is shown once with a copy button** (Section 9). It is not emailed.
3. Share the session link `/s/[slug]` with the community.
4. Access the dashboard at `/facilitator/[slug]` by entering the PIN.
5. Dashboard shows contribution count, breakdowns, most-common selections, and every individual vision.
6. Toggle session **visibility** (private ↔ public) and **status** (open ↔ closed).

### 5.3 Collective view
- When a facilitator sets a session **public**, contributors and visitors to `/s/[slug]` see a **mosaic** of all generated visions (Section 10).
- When **private**, only the facilitator (PIN) sees contributions; community members just see their own vision after submitting.

### 5.4 Progress bar mapping (important)
The form has **8 screens** but the progress bar shows **5 labeled steps**: `Scale · Values · Feelings · Features · Your vision`. Steps 2a/2b/2c all live under the **"Values"** step — the bar does **not** advance across the three sub-steps; instead the step label reads `Step 2a of 5`, `2b`, `2c` while the "Values" dot stays active. Keep this exact behavior.

## 8. Claude SVG generation (the core)

### 8.1 Endpoint
`POST /api/generate` (Next.js route handler, server-side).

**Request body:** the full contribution payload (scale, all selections, both open-text fields, myPeopleAre, displayName, language, optional sessionId).

**Response:** `{ success: true, id, svg, url }` or `{ success: false, error, code }`.

The route: validates input → assembles the prompt → calls Claude → strips any non-SVG wrapper (markdown fences, prose) → validates it's a single well-formed `<svg>` → persists the contribution (if Supabase configured) → returns the SVG + the new vision id/url.

### 8.2 SDK call
```ts
import Anthropic from '@anthropic-ai/sdk';
const client = new Anthropic(); // reads ANTHROPIC_API_KEY from env

const res = await client.messages.create({
  model: 'claude-sonnet-4-6',
  max_tokens: 4096,
  messages: [{ role: 'user', content: assembledPrompt }],
});
let svg = res.content.filter(b => b.type === 'text').map(b => b.text).join('');
svg = stripToSvg(svg); // remove ``` fences / preamble, keep <svg>…</svg>
```

### 8.3 Prompt assembly
Build one user message. Template:

```
You are a generative design tool for community visioning. From the inputs below,
produce ONE self-contained SVG that visualizes a community's dream {SCALE}.

Scale: {scale}
A 150-year vision of their people: "{myPeopleAre}"
How they want their body & senses to feel: {sensoryFeelings|comma list|"(none given)"}
Who and what belongs here: {communityBelonging|comma list|"(none given)"}
What they sense when they imagine being there: "{sensoryImagination|or "(none given)"}"
How it should feel emotionally: {feelings|comma list|"(none given)"}
Concrete features they want: {features|comma list|"(none given)"}
Their description in their own words: "{finalVision|or "(none given)"}"

STYLE: an architectural-collage hybrid — layered flat shapes that read as an
{SCALE}-level composition, abstract and interpretive, never literal or photoreal.
Hand-drawn vision-board feel. Add texture through pattern repetition (hatching,
dots, organic accents like plants, water, people silhouettes). Hopeful and warm.

SCALE GUIDANCE:
{scale-specific composition guidance — see 8.4}

PALETTE: terracotta/coral for built structure, forest green for nature/open space,
warm amber for community/light, dusty blue for sky/water/transit, cream/off-white
for light & openness. Use the listed feelings and senses to drive mood, color
balance, and density.

OUTPUT RULES:
- Output ONLY a valid <svg>…</svg>. No explanation, no markdown, no backticks.
- Use viewBox="0 0 800 600". Make it responsive (no fixed width/height in px).
- Keep it self-contained: inline shapes/paths/gradients only. No external refs,
  no <image>, no scripts, no foreignObject.
```

### 8.4 Scale-specific composition guidance
- **room:** close-up interior; layered shapes implying furnishings, a window of light, texture; intimate framing.
- **building:** facade-forward; floors, windows, a rooftop element; collage of overlapping forms with plants and figure silhouettes.
- **block:** street-level; buildings along a horizontal baseline; trees, benches, a mural, a crossing; foreground/background depth.
- **city:** aerial/isometric-inspired; abstract skyline and district blocks; green patches, transit lines, density gradients; geometric and map-like.
- **planet:** ecosystem scale; organic land/water/sky relationships; forests, coastline, atmosphere; flowing, less rectilinear.

### 8.5 Robustness
- **Sanitize before render.** Strip `<script>`, event handlers, `<foreignObject>`, and external URLs from the returned SVG before storing/rendering (defense in depth even though it's our own model). Render via a sanitized inline SVG, not `dangerouslySetInnerHTML` of raw model output without cleaning.
- **Validate** the result parses as XML and has a root `<svg>`. On failure, retry once; if still bad, return a friendly error with a "Try again" affordance.
- **Loading state:** calm SVG-based animation + message *"Building your vision…"* (i18n key). No spinner anxiety, no fake progress bar.
- **Timeouts/errors:** show *"We had trouble generating your vision. Please try again."* + retry button. Log server-side.

## 9. Sessions & facilitator

### 9.1 Creating a session  (`/facilitator/new`)
Fields: title (required), description (optional), scale mode (`fixed:<scale>` or `open` = participants choose), default visibility (`private` default). On submit:
- generate a unique URL `slug` (human-readable, e.g. `west-oakland-block`, collision-safe);
- generate a numeric/alphanumeric **PIN**, store only a **hash** (bcrypt/argon2) in the row;
- **show the PIN exactly once** on a confirmation screen with a **Copy** button and a clear warning that it won't be shown again. Do **not** email it.

### 9.2 Joining  (`/s/[slug]`)
- If session is **open/active**, a community member runs the normal flow; on generate, the contribution is saved with that `session_id`. If the session scale mode is `fixed`, skip Step 1 and use the fixed scale.
- If session is **closed**, show a gentle "This session is closed" state (still allow viewing the mosaic if public).

### 9.3 Dashboard  (`/facilitator/[slug]`)
- PIN gate (verify against stored hash; rate-limit attempts).
- Shows: total contributions; breakdown by scale; most-common sensory feelings, belonging tags, feelings, and features (frequency counts); list of every contribution with its SVG + inputs.
- Controls: **visibility toggle** (private ↔ public), **status toggle** (open ↔ closed). Both are "modify standing configuration" actions — make them explicit, with the current state always visible.

### 9.4 Editing window
Contributions are **editable until the session is closed**. Implement a lightweight edit affordance on the vision page (`/vision/[id]`) — re-open the form pre-filled, re-generate, and update the row — allowed only while the parent session status is `open`. A standalone (sessionless) contribution is editable by whoever holds its link while it exists. Once the session is `closed`, contributions are read-only.

## 10. Collective mosaic
- Rendered on `/s/[slug]` when the session is **public**.
- Responsive grid of contribution cards; each shows the SVG thumbnail, the contributor's display name or "Anonymous", the chosen scale, and up to 3 of their selected belonging/feeling tags.
- Clicking a card opens that vision (`/vision/[id]`).
- When the session is **private**, this view is replaced by a simple message that contributions are being collected privately.

## 11. Internationalization
- Scaffold `i18next` + `react-i18next` now. Put **all** user-facing strings (labels, questions, tags, buttons, errors, loading) in an `en` resource file — no hardcoded display strings in components.
- Language selector (globe) in the top bar; default from `navigator.language`, fallback `en`.
- **This scope ships English only.** Spanish and Mandarin resource files are out of scope here but the architecture must make adding them a content-only task (no refactor). Open-text answers are passed to Claude as-is in whatever language the user writes.

## 12. Data model (Supabase / Postgres)

```sql
create table sessions (
  id uuid primary key default gen_random_uuid(),
  slug text unique not null,
  title text not null,
  description text,
  scale_mode text not null default 'open',      -- 'open' | 'room'|'building'|'block'|'city'|'planet'
  visibility text not null default 'private',    -- 'private' | 'public'
  status text not null default 'open',           -- 'open' | 'closed'
  facilitator_pin_hash text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table contributions (
  id uuid primary key default gen_random_uuid(),
  session_id uuid references sessions(id) on delete cascade,  -- nullable for standalone
  display_name text,
  language text not null default 'en',
  scale text not null,                            -- room|building|block|city|planet
  my_people_are text,
  sensory_feelings text[] not null default '{}',
  community_belonging text[] not null default '{}',
  sensory_imagination text,
  feelings text[] not null default '{}',
  features text[] not null default '{}',
  final_vision text,
  svg_output text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index on contributions(session_id);
create index on sessions(slug);
```

**Access:** all writes go through server routes. If using Supabase RLS, lock down direct client access and use the service role only server-side. The browser never holds privileged keys.

## 13. Build order

1. **Scaffold** — Next.js + TS + Tailwind; design tokens (§6); `<Logo>`, font stack, layout shell; i18n scaffold with `en` strings.
2. **Form store + shells** — Zustand store (full `FormData`, step index, setters, caps for feelings≤5 / features≤8); `ProgressBar` (5 labels, 2a/2b/2c sub-labeling per §5.4); `NavigationButtons`.
3. **Screens** — build 7.0 → 7.7 to spec, wired to the store. Get the whole click-through working with no backend.
4. **`/api/generate`** — prompt assembly (§8.3–8.4), SDK call, strip+sanitize+validate (§8.5). Test against real output and iterate on prompt quality. **Budget real time here.**
5. **Vision page** `/vision/[id]` — render SVG, share link (copy), **PNG export** (SVG→canvas→PNG). PDF export optional.
6. **Supabase** — schema (§12), server data layer, persist on generate, fetch for vision/mosaic/dashboard.
7. **Sessions** — `/facilitator/new` (+ show-once PIN), `/s/[slug]` join, fixed-vs-open scale handling.
8. **Dashboard** `/facilitator/[slug]` — PIN gate, aggregates, contribution list, visibility + status toggles.
9. **Mosaic** — public `/s/[slug]` grid (§10).
10. **Edit window** — pre-filled re-generate while session open (§9.4).
11. **Pass** — a11y check (focus, labels, contrast, reduced-motion), mobile pass at 360–390px, error/empty/closed states.

Ship 1–5 as a satisfying standalone loop before layering 6–10.

## 14. Out of scope (do not build this round)
- Spanish / Mandarin translations (scaffold only).
- PDF export (PNG is required; PDF is a nice-to-have if time remains).
- Facilitator report export (ZIP of PNGs + CSV).
- User accounts / auth beyond the facilitator PIN.
- File uploads.
- Rate limiting / abuse protection on the public generate route (note it as a fast-follow; low-traffic MVP).
